# Product

> Scope: this file describes **OTP** (orgtp.com, source in `src/`). This repo
> also ships **Orger** (orger.ai, source in `orger/`), which has its own design
> system at `orger/DESIGN.md`. When working on Orger, that file governs, not this one.

## Register

brand

## Users

Founders and operators running, or about to run, an AI agent team. The person
with fourteen AI tools open across fourteen browser tabs: agents scattered
across scripts, crontabs, and chat windows, no shared structure, nobody able to
say what each agent is actually doing.

Their context when they reach OTP: they are scaling an AI-augmented org and the
coordination layer has not kept up. Senior people are drowning. SOPs written
last quarter are already ignored. Every new agent gets re-explained from
scratch. They are not shopping for another tool, they are looking for the
operating layer that makes the tools they already have run as one org.

The job to be done: give every seat, human or AI, a role, an SOP, a KPI, and an
accountability line, so the organization runs as a coordinated system instead of
a pile of tabs.

## Product Purpose

OTP (Organization Transport Protocol) is the coordination intelligence layer for
AI-native organizations. AI agents and humans hold seats on the same org chart.
SOPs are runtime context the agent inherits every session, not documents that
rot. KPIs roll up. Organizations publish their operating system (OOS) so
coordination intelligence compounds across the network the way knowledge
compounds on Wikipedia.

It exists because org charts were built for humans and break when agents arrive,
and because the founder hit that wall running his own agency and built the layer
he needed. Success looks like a founder mapping their chart, running their org
on OTP within a week, and the network of published operating systems growing.
The near-term goal is the first 50 founding signups.

## Brand Personality

Founder-direct and structural. The voice is first person, opinionated, and
honest, with dry confidence rather than hype. It speaks like a founder who built
the thing because he needed it, willing to be specific and to name names (the
chart on the home page is his real agency's chart, with real agents and real
KPIs). Plain, declarative sentences. No growth-hack tone, no anonymous-SaaS
neutrality.

Three words: **direct, structural, honest.**

Emotional goal for the visitor: calm clarity and a sense of control. They should
leave feeling "I can finally see the whole system, and this is under control."

## Anti-references

- **Cheerful startup pastel.** Rounded everything, soft gradients, friendly
  blobs, cute mascots, playful illustration. OTP is a serious operating layer,
  not a toy. Warmth comes from honesty and the founder's voice, not from cuteness.
- **Enterprise software grey.** Dense, corporate, joyless legacy B2B (IBM,
  Salesforce). Cluttered chrome, no point of view. OTP must have a spine and an
  opinion: legible, considered, and clearly authored by someone.

## Design Principles

1. **Practice what you preach.** OTP is run by an AI agent team. The site should
   feel like it was built by an org that actually operates this way. Use the
   real chart, real agents, real KPIs as proof, not stock illustration.
2. **Show the system, do not sell the dream.** Clarity over hype. Concrete
   artifacts (the chart, SOPs, the Coordination Score, the install snippet) earn
   belief faster than vision language. The visitor should be able to see how it
   works, not just be told it works.
3. **Founder on the record.** First person, opinionated, specific. The voice has
   an author and is willing to take a position. Never retreat into neutral
   SaaS copy.
4. **One buyer, one door.** Speak to the operator running agents. Do not dilute
   a page trying to address coaches and publishers at the same time. Other
   audiences get their own surfaces.
5. **Structure is the brand.** OTP is literally a protocol and an org chart.
   Order, hierarchy, legibility, and the sense that everything has a seat should
   be felt in the design itself, not only described in the copy.

## Accessibility & Inclusion

Target WCAG 2.2 AA. AA contrast ratios on all text and meaningful UI, full
keyboard navigation with visible focus states, semantic landmarks, and
`prefers-reduced-motion` support for every non-essential animation. This is the
baseline for a public B2B site and is not optional.
